# Room Booking Service

Сервис бронирования переговорок. REST API на Go с JWT-авторизацией, PostgreSQL и Docker.

## Быстрый старт

```bash
docker compose up --build
```

Swagger документация к апи доступна по: `http://localhost:8080/swagger/`


## Схема базы данных

| Таблица | Ключевые поля |
|---------|---------------|
| `users` | id (UUID), email (уникальный), password_hash, role, created_at |
| `rooms` | id (UUID), name, description, capacity, created_at |
| `schedules` | id (UUID), room_id (уникальный FK), days_of_week (INTEGER[]), start_time, end_time |
| `slots` | id (UUID), room_id (FK), start_time (TIMESTAMPTZ), end_time (TIMESTAMPTZ) |
| `bookings` | id (UUID), slot_id (FK), user_id, status (active/cancelled), conference_link, created_at |

Индексы:
- `slots(room_id, start_time)` - ускоряет поиск слотов по переговорке и дате
- `UNIQUE INDEX ON bookings(slot_id) WHERE status = 'active'` - гарантирует одну активную бронь на слот



## Принятые решения

### Формирование слотов

Выбран подход скользящего окна на 7 дней вперёд

При запросе доступных слотов система проверяет, покрывает ли текущий набор слотов в БД близжайшие 7 дней от сегодняшней даты. Если нет - генерирует недостающие слоты на лету и сохраняет их в БД. Я использовал этот подход потому что в условии задачи сказано, что 99,9% случаев используют слоты в пределах ближайших 7 дней. Более того, так как это самый горячий эндпоинт, генерировать слоты во время запроса будет превышать ориентир в 200ms.
Тажке, в качестве альтернативы можно использовать anacron джобу на сервере, на котором будет запускаться сервис, которая будет обновлять данные о слотах. Еще в качестве оптимизации можно добавить in-memory кеш, чтобы не отправлять дополнительные запросы в бд.

### Предотвращение конфликта бронирования

Конфликт исключается на уровне БД с использованием уникального индекса по активным броням:

```sql
CREATE UNIQUE INDEX bookings_active_slot_uidx ON bookings(slot_id) WHERE status = 'active';
```

Это гарантирует, что на один слот может быть только одна активная бронь - даже при конкурентных запросах, без дополнительных блокировок на уровне приложения.
